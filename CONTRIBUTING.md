# Contributing to Termin Conformance

Thanks for your interest in contributing. This repository holds the
authoritative spec and test suite for any conforming Termin runtime. Most
contributions fall into one of three categories:

1. **New conformance tests** — expanding coverage of the contract.
2. **Spec clarifications** — improving the Runtime Implementer's Guide,
   IR JSON Schema documentation, or package format spec.
3. **Bug reports** — from runtime implementers who hit a case the suite
   doesn't yet cover or where the suite is wrong.

If you're building a non-reference Termin runtime, the [conformance
suite README](README.md) explains how to implement an adapter and run
the tests against your runtime. Bug reports against the suite itself
are welcome.

## Getting started

1. **Read [termin.dev](https://termin.dev)** first to understand the
   project's scope and guarantees.
2. **Read the [Runtime Implementer's Guide](specs/termin-runtime-implementers-guide.md)**
   to understand what the conformance suite is specifying.
3. **Open an issue before writing code** for anything non-trivial.
   Particularly for new test categories or spec changes: a short
   problem description is much easier to review than a large PR.
4. **Fork, branch, and develop.** Standard GitHub flow.

## Developer Certificate of Origin (DCO)

Every commit must be signed off to certify you have the right to
contribute it under the project's license. This is the [Developer
Certificate of Origin](https://developercertificate.org), the same
mechanism used by the Linux kernel, Docker, and many other
open-source projects. We do not use a CLA.

By signing off your commits, you certify the DCO text. Summary: you
wrote the code (or test, or spec change), or you have the right to
contribute it, and you agree it can be distributed under the project's
Apache 2.0 license.

### How to sign off

Add a `Signed-off-by:` trailer to every commit message. Git can do
this for you automatically with the `-s` flag:

```bash
git commit -s -m "Add conformance test for compound verb access grants"
```

That produces a commit message like:

```
Add conformance test for compound verb access grants

Signed-off-by: Alex Doe <alex@example.com>
```

Use your real name (or the name you publish under) and a real email
address. GitHub-generated `noreply` addresses are acceptable.

If you forget the sign-off, add it to your most recent commit:

```bash
git commit --amend --signoff
```

Or to a range of commits:

```bash
git rebase --signoff HEAD~3
```

The DCO check runs on every pull request. PRs without sign-offs on
every commit are not mergeable.

## Test discipline

- **Tests must pass against any conforming runtime, not just the
  reference.** If a test only passes against `adapter_reference.py`,
  it's testing implementation, not contract. Avoid assumptions about
  implementation details (specific error messages, rendering order,
  internal paths) that a different runtime might legitimately
  implement differently.
- **Browser tests use `data-termin-*` selectors, never literal
  English text.** A user changing a button label from "Delete" to
  "Remove" must not break conformance. Marker attributes are the
  canonical affordance.
- **Use the `_require_served_url` helper** for tests that need a real
  HTTP URL (e.g., Playwright navigation) rather than branching on
  adapter name. Tests should not know which adapter is running.
- **Run the full suite before submitting.** At minimum
  `TERMIN_ADAPTER=reference pytest tests/ -v` must pass with zero
  failures. Browser tests (`TERMIN_ADAPTER=served-reference` +
  Playwright) are strongly recommended.
- **New test categories belong in a dedicated file.** Version-specific
  additions go into `test_v0X_*.py`; topical additions go into the
  corresponding `test_*.py` (access_control, state_machines, crud,
  etc.).

## Fixture discipline

The fixtures under `fixtures/` are **generated artifacts**:

- `fixtures/*.termin.pkg` — compiled application packages
- `fixtures/ir/*.json` — extracted IR JSON
- `fixtures/*.deploy.json` — deploy configs for agent/channel apps

**Do not hand-edit these files.** They are regenerated by
`util/release.py` in the [termin-compiler](https://github.com/jamieleigh3d/termin-compiler)
repo and will be overwritten on the next release. If a fixture is
wrong, fix the source `.termin` file in the compiler repo's
`examples/` directory (or the compiler itself), regenerate, and
submit the corrected fixture alongside the source change.

## Spec changes

Changes to the specifications (`specs/termin-ir-schema.json`,
`specs/termin-runtime-implementers-guide.md`,
`specs/termin-package-format.md`) affect every runtime implementer.
They require:

1. **A clear motivating case** — what behavior needs specifying, and
   what goes wrong today without the spec change.
2. **A corresponding conformance test** that would fail before the
   spec change and pass after.
3. **A CHANGELOG entry** describing the change for runtime authors.

Spec changes bump the IR version or package-format version per the
versioning scheme in the Runtime Implementer's Guide.

## Pull request checklist

Before marking your PR ready for review:

- [ ] Every commit is signed off (`Signed-off-by:` trailer)
- [ ] `TERMIN_ADAPTER=reference pytest tests/ -v` passes locally
- [ ] Browser tests pass, or are correctly skipped
  (`TERMIN_ADAPTER=served-reference` + Playwright installed)
- [ ] New tests use `data-termin-*` selectors rather than literal text
- [ ] No hand-edits to files under `fixtures/`
- [ ] Spec changes include a conformance test demonstrating them
- [ ] Changes that affect the contract include a `CHANGELOG.md` entry

## Scope of contributions

Especially welcome:

- Conformance tests for behaviors that are currently under-specified
- Non-reference runtime implementations (they live in their own
  repos; we're interested to hear from you)
- Bug reports with minimal repro cases, including the adapter and
  platform you hit them on
- Documentation improvements to the Runtime Implementer's Guide
- Platform-compatibility fixes (the suite is developed on Windows
  and Linux; macOS fixes welcome)

Out of scope:

- Proposals that tie the spec to a specific implementation detail
  of the reference runtime
- Commercial features (paid tiers, license-gated capabilities, etc.)
- Changes to `fixtures/` that bypass the termin-compiler regeneration
  path

## Security issues

Do not open public issues for security vulnerabilities in the test
suite or reference adapter. Email security@termin.dev for private
disclosure, mirroring the termin-compiler policy. See the compiler
repo's [SECURITY.md](https://github.com/jamieleigh3d/termin-compiler/blob/main/SECURITY.md)
for the full process.
